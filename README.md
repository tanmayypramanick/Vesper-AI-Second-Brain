## рҹӨ” Why does this exist?

Every AI assistant you talk to is completely stateless.

Ask it what you were working on last Tuesday вҖ” it doesn't know.  
Ask about someone you've been texting вҖ” it doesn't know.  
Ask it to recall that email from last week вҖ” it doesn't know.

Cloud AI (ChatGPT, Siri, Google Assistant) either stores nothing useful or stores *everything* on their servers вҖ” training on your private conversations to improve their products.

**Vesper is an attempt at something different:**

| | Cloud AI | Vesper |
|---|:---:|:---:|
| Reads your messages & emails | вң— | вң… |
| Knows your calendar | вң— | вң… |
| Knows who people in your life are | вң— | вң… |
| Remembers what you were doing last Tuesday | вң— | вң… |
| Data stays on your hardware | вң— | вң… |
| Works fully offline | вң— | вң… |
| Costs per query | рҹ’ё | вң— |
| Gets smarter about *you* over time | вң— | вң… |

The result feels less like a search engine and more like an assistant that's been quietly paying attention to your life for months.

### вҡ пёҸ The Maxwell Problem

The MX130 is Maxwell architecture (sm_5.0). Nearly every modern CUDA AI library requires Pascal (sm_6.0+). This constraint shaped every single model choice in the stack:

| Library | Needs | MX130 Result |
|---------|-------|-------------|
| Kokoro TTS *(wanted)* | sm_6.0+ | `cuDNN EXECUTION_FAILED` вқҢ |
| CTranslate2 Whisper CUDA | sm_6.0+ | crashes on load вқҢ |
| PyTorch CUDA general | sm_6.0+ | limited вқҢ |
| Ollama (llama.cpp) | sm_5.0+ | вң… works |

So: LLM inference uses Ollama on GPU. TTS and STT run entirely on CPU.

## рҹ§  Memory вҖ” How ChromaDB Works

[ChromaDB](https://www.trychroma.com/) is an open-source vector database. Every piece of data ingested into Vesper becomes a *memory* вҖ” a document stored with its 768-dimensional embedding vector.

When you ask Vesper a question:

```
Your question
     в”Ӯ
     в–ј  nomic-embed-text вҶ’ 768-dim vector
     в”Ӯ
     в–ј  Cosine similarity search across 12,000+ memory vectors
     в”Ӯ
     в–ј  Most semantically relevant memories returned
     в”Ӯ
     в–ј  LLM synthesises a response using that context
```

This is why *"what did my friend say about the concert?"* finds the right WhatsApp message вҖ” even if you didn't use the exact words from the original message.

### Memory Schema

Every memory is stored with:

```python
{
    "document": "the actual text content",
    "metadata": {
        "category":  "whatsapp" | "email_received" | "screen_ocr" | "contact" | ...,
        "source":    "whatsapp:PersonName" | "screen:AppName" | ...,
        "timestamp":  1748000000   # unix timestamp вҖ” enables "what happened last Tuesday?" queries
    }
}
```

### Single-Writer Constraint

ChromaDB's `hnswlib` C++ index segfaults under concurrent writes from multiple processes. Vesper enforces strict single-writer architecture вҖ” only `file_receiver.py` ever touches the database. A `threading.Lock()` inside it serialises every read and write. All ingest scripts are HTTP clients only.

## рҹӨ– AI Model Stack

### Language Models

| Model | Ollama tag | Used for | GPU layers | Speed |
|-------|------------|----------|-----------|-------|
| **dolphin-phi 2.7B** | `dolphin-voice` | All voice queries | 22/32 on MX130 | ~15 tok/s |
| **Qwen3 4B** | `qwen3-ask` | WhatsApp `/model qwen3` | CPU only | ~5 tok/s |

**Why dolphin-phi?** Based on Microsoft Phi-2 вҖ” small, fast, uncensored (no refusals on personal questions about relationships or private life), and partially fits on the MX130 GPU at 22/32 layers. Fast enough for real conversation.

**Why Qwen3 for WhatsApp?** Better reasoning and a 32K context window vs Phi-2's 2K. Used only on demand via `/model qwen3` where latency matters less than quality.

### Embeddings

| Model | Runs on | VRAM | Always loaded? |
|-------|---------|------|---------------|
| `nomic-embed-text` (768-dim) | GPU | 555 MB | Yes вҖ” never evicted from memory |

Every ChromaDB query and every ingest call goes through this. ~50вҖ“80ms per embed.

### рҹҺӨ Speech-to-Text

| Model | Mode | Real-time factor | Used for |
|-------|------|-----------------|---------|
| `distil-whisper-small.en` (int8) | CPU | 0.74x | Live kiosk queries, WhatsApp voice notes |
| `whisper-medium` | CPU | ~1.8x | Night batch вҖ” higher quality re-transcription of the day's audio |

RTF 0.74x means a 3-second clip takes ~2.2 seconds to transcribe. Can't use GPU (Maxwell/CTranslate2 incompatibility).

### рҹ”Ҡ Text-to-Speech

| Model | Mode | Latency per sentence |
|-------|------|---------------------|
| **Piper** `hfc_female-medium` / `ryan-high` | CPU | 62вҖ“220ms |
| Supertonic-3 | CPU | 330вҖ“830ms (fallback) |

Kokoro TTS was the target (near-human quality, ~50ms on GPU) but fails with `cuDNN EXECUTION_FAILED` on Maxwell. Piper is the best CPU-native option available.


## рҹҺҷпёҸ Voice Pipeline

**Latency journey: 25 seconds вҶ’ 1.5 seconds**

### End-to-End Flow

```
"Hey Vesper"
     в”Ӯ
     в”ңв”Җ Fast-whisper (wake word detection, continuous mode, lightweight)
     в”Ӯ
     в”ңв”Җ VAD detects speech start  (RMS threshold > 0.015)
     в”Ӯ
     в”ңв”Җ MediaRecorder starts  (WebM/Opus)
     в”Ӯ
     в”ңв”Җ POST /voice_prepare  вҶҗ fires while user is still speaking
     в”Ӯ     вҶ’ server pre-embeds the partial transcript
     в”Ӯ     вҶ’ ChromaDB results cached before user finishes talking
     в”Ӯ
     в”ңв”Җ 1.4s of silence вҶ’ recording stops
     в”Ӯ
     в”ңв”Җ Audio blob вҶ’ base64 вҶ’ POST /voice_fast
     в”Ӯ     вҶ’ Whisper STT     (~2s for a 3s clip on CPU)
     в”Ӯ     вҶ’ nomic-embed     (cache hit from /voice_prepare)
     в”Ӯ     вҶ’ ChromaDB search
     в”Ӯ     вҶ’ Query routing   (see table below)
     в”Ӯ     вҶ’ dolphin-phi generates response
     в”Ӯ
     в””в”Җ At every comma / period вҶ’ Piper TTS fires on that chunk immediately
           вҶ’ SSE stream chunks:
               data: T:<base64_text>   вҶҗ subtitle displayed instantly
               data: A:<base64_wav>    вҶҗ AudioContext plays the chunk
           вҶ’ First audio plays < 1 second after LLM starts generating
```

### рҹҡҰ Query Routing вҖ” First Match Wins

| Path | When it fires | Latency |
|------|--------------|---------|
| вҡЎ **Fast path** | Time, identity, greetings, weather, memory count | < 50ms вҖ” zero LLM, zero ChromaDB |
| рҹ—ғпёҸ **Data lookup** | Last message from X, calendar, contacts, recent screen activity | 200вҖ“800ms вҖ” ChromaDB only, no LLM |
| рҹ§  **LLM synthesis** | Complex personal queries needing reasoning | 1.5вҖ“2.5s вҖ” ChromaDB + dolphin-phi |
| рҹҡ« **Anti-hallucination guard** | No relevant memories found for a personal query | Explicit refusal вҖ” never guesses |

### рҹ’Ў The Key Optimisation: Comma-Split TTS

Instead of waiting for the full LLM response to finish generating before starting speech synthesis, Vesper fires Piper TTS at every natural pause вҖ” comma, period, semicolon. First audio chunk plays ~600ms after the LLM starts.

**Effect: 5.3s вҶ’ 2.0s** for memory-lookup queries.

### рҹ“ү Latency Evolution

| Optimisation | First-word latency |
|-------------|-------------------|
| Baseline вҖ” Qwen3 4B, CPU only | 9вҖ“25s |
| dolphin-phi, 22/32 layers on GPU | 3.5s |
| Shorter system prompt | 2.8s |
| Comma-split TTS streaming | 2.0s |
| Fast paths (time / greetings / identity) | < 0.5s |
| `/voice_prepare` pre-computation | **1.5вҖ“1.7s** |

## рҹ’¬ OpenClaw вҖ” WhatsApp Bot

Named after the unofficial WhatsApp Web client it's built on: [Baileys](https://github.com/WhiskeySockets/Baileys) вҖ” a Node.js implementation of the WhatsApp Web binary protocol. No official WhatsApp API supports automated access on personal numbers.

### Two-Bot Setup

**`bot.js`** вҖ” Indian number, personal WhatsApp (619 lines)
- Silently ingests every received message into ChromaDB
- `OWNER_JID` gate вҖ” only the US number gets LLM responses. Everyone else (family, friends) is ingested silently. *(This gate was added after the bot replied "Yeah, I'm right here!" into the middle of a personal conversation.)*
- Runs a small HTTP server on port 5001 for outbound messages (used by `morning_briefing.py` and `proactive_alerts.py`)

**`bot_us.js`** вҖ” US WhatsApp Business number (140 lines)
- Ingest-only. Never replies to anyone.
- Captures all US-side conversations вҖ” professional contacts, etc.

### What It Can Handle

| Message type | What Vesper does |
|-------------|-----------------|
| рҹ’¬ Text | Answers via LLM with memory context |
| рҹҺӨ Voice note | Downloads вҶ’ Whisper вҶ’ LLM answer |
| рҹ“ё Photo | LLM vision вҶ’ describes + stores as memory |
| рҹ”— URL / article link | Fetches the page вҶ’ summarises |
| `"remember [fact]"` | Regex match вҶ’ stores directly to ChromaDB |
| `/model qwen3` | Switches to Qwen3 (smarter, 32K context, slower) |
| `/model dolphin` | Switches back to dolphin-phi (fast) |
| `/clear` | Clears the conversation history for this session |

Conversation history (last 6 turns) is persisted per-JID in `history.json` вҖ” multi-turn conversations work across restarts.

## рҹ‘ҒпёҸ Screen Activity вҖ” `vesper_capture.py`

Uses Apple's native **Vision framework** (via PyObjC) for OCR вҖ” the same engine macOS uses for Live Text:

1. Watches `NSWorkspace` application activation events
2. On each app switch or significant window change, takes a screenshot
3. Runs Apple Vision OCR (faster than Tesseract, runs on Apple Neural Engine)
4. Strips UI chrome, menus, and navigation noise via heuristics
5. POSTs `{app_name, ocr_text, timestamp}` to `/store_ocr`

Vesper now has a fully searchable record of every app, document, code file, article, and video you engaged with вҖ” stored as `screen_ocr` memories tagged with the app name.

## рҹ–јпёҸ The 3D Orb вҖ” `kiosk.html`

A single self-contained HTML file, 1,360 lines. Zero build step. Zero npm. Served by Flask at `/kiosk`.

Built with **Three.js r128** and a custom GLSL fragment shader:

- Inner sphere with procedural sine-wave displacement simulating an ocean surface
- Deep navy blue base with cyan rim glow вҖ” shifts to green during listening, white burst on wake
- Wireframe grid overlay at 9% opacity
- 220-particle cloud orbiting the sphere
- Equatorial glow ring
- 0.6Hz breathing pulse (scale oscillates 1.0вҖ“1.012)
- **Audio-reactive** вҖ” RMS of live mic input scales the orb in real-time during recording

### State Machine

```
idle в”Җв”Җв–ә listening в”Җв”Җв–ә thinking в”Җв”Җв–ә responding в”Җв”Җв–ә idle
```

Each state change updates orb colour/animation intensity, shows/hides subtitle text, and manages AudioContext scheduling.

### Always-On Display (AOD)

After 10 minutes idle вҶ’ pure black background, large ultra-thin clock, breathing 48px orb, soft pulsing hint. Wake word detection continues running through AOD.

## рҹҢҗ Remote Access вҖ” Tailscale

Tailscale creates a WireGuard-based mesh VPN. The server gets a stable IP and hostname (`vesper-server.tail614590.ts.net`).

- Access Vesper from anywhere вҖ” not just home LAN
- The Nord kiosk switches from LAN IP to Tailscale hostname when off-network
- SSH from anywhere: `ssh tanmay@100.x.x.x`
- Tailscale issues a **trusted TLS certificate** for the server hostname вҖ” no self-signed cert warning on remote devices

Cron renews the cert monthly: `tailscale cert <hostname>` + server restart.

## рҹ—әпёҸ `file_receiver.py` вҖ” Route Map

The entire server in one file. Flask HTTPS on port 5000. 1,645 lines. 22 routes. The only process that writes to ChromaDB.

| Route | Method | What it does |
|-------|--------|-------------|
| `/health` | GET | `{"status":"ok","memories":N}` вҖ” polled by watchdog |
| `/system_stats` | GET | Live CPU, RAM, GPU util/VRAM/temp, disk, uptime |
| `/ingest` | POST | Async memory write вҖ” 202, queued in ThreadPoolExecutor |
| `/store_memory` | POST | Sync memory write вҖ” blocks until ChromaDB confirms |
| `/store_ocr` | POST | Screen OCR storage with `app_name` metadata |
| `/voice_fast` | POST | Main voice endpoint вҖ” SSE stream of text + audio chunks |
| `/voice_prepare` | POST | Pre-embeds partial transcript while user is still speaking |
| `/transcribe` | POST | Base64 WAV вҶ’ Whisper text |
| `/transcribe_partial` | POST | Streaming partial transcript for live kiosk subtitle |
| `/ask` | POST | WhatsApp deep-query via Qwen3 (90s timeout) |
| `/recall` | POST | Raw ChromaDB query, no LLM |
| `/ambient_context` | GET | Recent screen activity for kiosk sidebar |
| `/kiosk` | GET | Serves `kiosk.html` |
| `/ui` | GET | Serves `voice_ui.html` |

**Threading model:**
- `_chroma_lock` (`threading.Lock`) вҖ” all ChromaDB reads and writes, one at a time
- `_llm_sem` (`threading.Semaphore(1)`) вҖ” one LLM inference at a time in `/ask`
- `_executor` (`ThreadPoolExecutor(max_workers=2)`) вҖ” async `/ingest` queue

## рҹҡ§ Known Issues & Limitations

| Issue | Why | Workaround |
|-------|-----|-----------|
| No iPhone screen capture | iOS blocks background OCR API access for third-party apps | Send screenshots to Vesper via WhatsApp |
| WhatsApp Baileys ToS risk | Unofficial protocol вҖ” Meta could block the session | Manual export fallback via `export_whatsapp.py` |
| GPU = Maxwell sm_5.0 | Blocks Kokoro TTS, CTranslate2 Whisper, modern CUDA libs | Accepted вҖ” CPU fallback for TTS + STT |
| iMessage 15-min lag | No real-time hook into Messages.app | `imessage_realtime.sh` gets it to ~30s |
| ChromaDB single-writer only | hnswlib segfaults under concurrent writes | Enforced by architecture вҖ” `_chroma_lock` |

| Audio memories have overnight lag | Night batch runs at 2 AM | Real-time audio indexing is on the roadmap |
| SSL cert on Android | Self-signed cert requires manual install | Tailscale cert removes this for remote access |

### рҹҺ§ Wearable Audio Diary

A tiny wearable microcontroller (targeting Seeed XIAO nRF52840 вҖ” already have the board) worn on the body. Records ambient audio continuously. Clips stream over BLE to the server where Whisper transcribes, pyannote diarizes speakers, and memories are stored with context tags.

Full pipeline: **speaker diarization** (who said what?) вҶ’ **context classifier** (meeting / phone call / background TV / private?) вҶ’ **semantic chunking** вҶ’ ChromaDB `audio_diary` category.

> *"What did I say about the project on Monday?"*
> *"Who called me yesterday?"*
> *"Summarise my conversations this week."*

`vesper_audio.py` on the Mac is already the desktop version of this вҖ” it records 24/7 in 30s chunks and re-transcribes overnight with `whisper-medium`. The wearable extends that to anywhere.
